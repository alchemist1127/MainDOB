# PCIe / modern-hardware porting — work-in-progress status

Target: bring MainDOB from the Compaq Armada E500 (PCI, PIIX4, 8259) toward
the Acer Extensa 5220 (PCIe, ICH8-M, APIC). This document records what was
implemented in the porting session, what remains, and — importantly — what
has **not** been tested yet.

The guiding invariant throughout: **dual compatibility**. Every change is
additive and opt-in. On hardware that boots today (QEMU `pc`, Armada E500)
each path takes the legacy branch, byte-for-byte as before. The new code
activates only when the hardware offers the feature.

## CRITICAL: nothing here has been built or run

All code was written and syntax-checked with a native `gcc` in freestanding
32-bit mode (`-Wall -Wextra`, zero warnings), and the Multiboot header was
verified by computing its bytes by hand. **None of it was compiled with
`i686-elf-gcc`, assembled with `nasm`, or booted in QEMU** — the work
environment had no cross-toolchain, no nasm, and no network.

Five stacked, unverified changes now sit on critical kernel code (PCI
config access, the Multiboot header, and the entire interrupt path). The
first real test — `make` plus a QEMU boot — is the meaningful checkpoint and
should happen before any driver work builds on top of this. Treat the
interrupt-stack code especially as "plausible but unproven".

## What was implemented

### 1. PCIe configuration access (ECAM) — kernel mechanism

- New `kernel/acpi/` module. Scans for the ACPI RSDP (EBDA + 0xE0000 area),
  walks RSDT/XSDT, extracts MCFG (ECAM base + bus range) and MADT (LAPIC,
  IOAPICs, interrupt source overrides). No AML interpreter — static tables
  only.
- `sys_pci_read` / `sys_pci_write` became dual-mode: an ECAM branch in
  front (memory-mapped config, offsets 0..4092, reaches extended
  capabilities), the legacy `0xCF8/0xCFC` path unchanged behind it.
- `boot/hotplug/main.c` now reads PCI config through `pci_config_read`
  (the kernel syscall) instead of poking the ports directly, so its
  enumeration inherits ECAM and can see extended config space.
- A `pci_legacy` force-legacy override exists in the ACPI module
  (`acpi_force_legacy_pci`) but is **not yet wired to a boot tag** — the
  kernel does not parse its own Multiboot command line today. Wiring it is
  a small, isolated change in `kernel/boot/boot_info.c` and is strongly
  recommended before the first real-hardware boot (it lets you fall back to
  PIC/legacy without recompiling if ECAM/IOAPIC misbehaves on the Extensa).

### 2. Explicit VGA text mode request

- `kernel/boot/entry.asm` Multiboot header extended with the video flag
  (bit 2) and `mode_type=1` (text). Header arithmetic verified by hand:
  magic+flags+checksum ≡ 0 mod 2^32, video fields at offset 32. The
  loader now puts the machine in VGA text mode (best-effort per spec)
  before the kernel runs, so the `0xB8000` console is valid by
  construction on CSM-era BIOSes. Not guaranteed on a hypothetical
  text-mode-less BIOS; the fallback (a framebuffer console rendering
  fonts) is deliberately deferred until proven necessary.

### 3. Interrupt-controller abstraction (Phase 0)

- `kernel/arch/x86/intr.{h,c}`: a vtable (`intr_controller_t` —
  mask/unmask/eoi/is_spurious), a per-vector handler table (256, replacing
  the old per-line table of 16), event-driven vector allocation for the
  0x50..0xDF range (allocated on driver request, freed on process death —
  no polling), and a unified dispatcher `intr_dispatch`.
- `kernel/arch/x86/isr_stub.asm`: a NASM `%rep` loop generates 144 generic
  per-vector trampolines (`vec_stub_0x50..0xDF`). `irq_common_stub` now
  calls `intr_dispatch`. `kernel/arch/x86/vec_stubs.h` (generated by
  `tools/gen_vec_stubs.py`) declares them and builds the address table.
- `kernel/arch/x86/irq.c` refactored: the 8259 logic became the **PIC
  backend** (`pic_controller`), installed as the default at `irq_init`.
  The legacy line API (`irq_register_handler` etc.) still works, now
  translating line L → vector 32+L. With only the PIC backend installed,
  behaviour is identical to before the abstraction.

### 4. IRQ sharing on the PIC (Phase 1, strategy C)

- One PIC line can now be shared by several drivers (PCIe INTx). On fire,
  the kernel notifies all registered sharers; each inspects its own
  device, services it if it was the source, and calls `irq_done`
  unconditionally. A refcount of expected `irq_done`s drives a collective
  unmask; a one-shot safety timer (50 ms, via the tickless timer
  subsystem) force-unmasks if a buggy driver never reports, avoiding a
  permanent mask.
- Single-sharer lines (timer, keyboard, ATA, any unshared PCI line)
  collapse to exactly the old mask-on-fire / unmask-on-done behaviour.
  Existing drivers (verified against AHCI) need no changes — they already
  call `irq_done` once per fire.

### 5. IOAPIC (Phase 2)

- `kernel/arch/x86/ioapic.{h,c}`: discovers and maps every IOAPIC from the
  MADT (uncached MMIO), reads IOAPICVER for input count, masks all inputs,
  and provides `ioapic_route_gsi` / mask / unmask plus an `ioapic_controller`
  vtable whose EOI goes to the LAPIC.
- `intr_switch_to_ioapic()` (in intr.c) programs a redirection entry for
  each of the 16 legacy lines (GSI resolved via MADT overrides, so
  IRQ0→GSI2 for the PIT is honoured), masks the 8259 entirely, makes the
  IOAPIC the active controller, and re-arms lines that had handlers.
  `intr_line_mask/unmask` route through whichever controller is active.
- Called from `kmain` only when `ioapic_init()` succeeds. Verified that the
  LAPIC timer is independent of PIC/IOAPIC (own vector 0xF0, own EOI), so
  masking the PIC during migration cannot stop the scheduler tick.
- PCI device routing uses **Option A**: rely on the BIOS-programmed IRQ
  line register (config 0x3C), which already flows from hotplug to drivers.
  No bridge-recursion INTx swizzling and no `_PRT`/AML — sufficient for
  CSM-era hardware (Extensa included), insufficient only for UEFI-pure
  systems that configure IRQs at runtime via ACPI (out of target).

### 6. MSI (Phase 3)

- `sys_msi_enable` (syscall 107): driver passes bus/slot/func + the MSI
  capability offset it found by walking the capability list; the kernel
  allocates a vector, computes the x86 Message Address (0xFEE00000, dest
  LAPIC 0) and Data (vector, fixed, edge), handles 32/64-bit layout,
  programs the device, installs a handler, sets MSI Enable.
- MSI is edge-triggered and dedicated: no masking, no refcount, simple
  notify relay (`msi_forward_handler`). The dispatcher EOIs allocatable-
  range vectors (0x50..0xDF) to the LAPIC always, independent of the
  active legacy-line controller.
- Teardown clears a dead process's MSI notify ports before releasing its
  vectors. libc wrapper: `msi_enable(...)`.
- **MSI only, not MSI-X.** MSI-X (cap 0x11, table in BAR MMIO, many
  vectors) is a future extension; the ICH8 HD Audio target uses MSI.

## Suggested first-boot checklist (when toolchain is available)

1. `make` with `i686-elf-gcc` + `nasm`. Watch for the `%rep` trampoline
   loop assembling and the extended Multiboot header being accepted by GRUB.
2. Boot QEMU `pc` (no ECAM, no IOAPIC): confirm the same self-tests pass and
   timer / keyboard / ATA behave as before — this exercises the PIC backend
   through the new abstraction and the single-sharer IRQ path. This is the
   regression test that matters most.
3. Boot QEMU `q35` (has ECAM + IOAPIC): expect the ACPI log to show MCFG and
   MADT lines, `[INTR] Migrated to IOAPIC delivery`, and devices still
   working. This exercises ECAM, the IOAPIC backend, and migration.
4. Wire the `pci_legacy` boot tag before touching real hardware, so the
   Extensa can be brought up in legacy mode first and ECAM/IOAPIC enabled
   incrementally.

## QEMU test configuration (this session)

The Makefile was switched to exercise the new code paths:

- **Machine: `q35`** (was the implicit `pc`). q35 provides an ECAM region
  (MCFG) and an IOAPIC (MADT), so the kernel takes the memory-mapped PCI
  config path and the IOAPIC backend + migration instead of the legacy
  ports and the 8259. Selected via `QEMU_MACHINE` near the top of the
  Makefile; set it to `$(QEMU_MACHINE_PC)` to reproduce the old legacy
  machine for regression testing.
- **Disk: SATA/AHCI** (`QEMU_HDA = QEMU_HDA_SATA`), an explicit `ich9-ahci`
  controller with the disk on `ahci0.0` (driver: ahci, shows "SATA"). This
  port works. A `QEMU_HDA_Q35` alias for the chipset's built-in AHCI is
  provided as an alternative but is not the default.
- **Optical drives: `-cdrom` / auto-attach** on q35, NOT an explicit
  ich9-ahci port index. Attaching to `bus=ahciN.M` for M>0 turned out not
  to be portable — the target QEMU rejected both `ahci0.2` and `ahci0.1`
  with "Bus not found", even though the controller's own port scan reports
  ports 0..5. The register-level AHCI port count and QEMU's command-line
  `-device` bus topology are different things; only port .0 of an
  explicitly-created ich9-ahci is reliably addressable. So the boot CD uses
  the plain `-cdrom` shorthand (the form that booted in the earliest runs)
  and the empty/secondary optical drives use `ide-cd` with no `bus=`,
  letting QEMU place them on the machine's built-in AHCI. The boot CD may
  therefore sit on a different AHCI controller than the disk; that is fine
  now that the AHCI driver stays resident and does runtime hotplug (it no
  longer needs disk and optical on one controller in one instance).
- The duplicate `-machine type=pc` that was buried in the audio flags was
  removed (a second `-machine` would conflict); the AC97 codec attaches on
  q35 the same way.
- CD-ROM attachment was made machine-aware (`QEMU_CDROM_EMPTY` /
  `QEMU_CDROM_ISO`): q35 has no legacy IDE bus, so optical drives now hang
  off the AHCI controller (`ide-cd` on `ahci0.1`) instead of
  `if=ide,index=2`.
- The **boot ISO** (the live CD that runs the installer, used by `make run`
  / `debug` / `re-run` / `run-cd`) is attached **explicitly** to the AHCI
  controller on q35 (`ahci0.2`, `bootindex=0`) rather than via the bare
  `-cdrom` shorthand. The install workflow is `make run` (boot live CD →
  installer wizard writes the disk) then `make run-disk` (boot the installed
  disk), so the CD path is load-bearing, not incidental — leaving it to
  QEMU's non-deterministic auto-attach on q35 was a real risk and is now
  removed. On pc the legacy `-cdrom` form is kept.

AHCI port map on q35 (controller `ahci0` from `QEMU_HDA_SATA`):

| Port      | Use                        | Target(s)            |
|-----------|----------------------------|----------------------|
| `ahci0.0` | installed disk             | all                  |
| `ahci0.1` | empty / secondary optical  | `run-disk`, `run-cd` |
| `ahci0.2` | boot ISO (live CD)         | `run`, `debug`, `run-cd` |

Ports are distinct so they coexist without collision in every target.

## Post-test fixes

After the first build/run attempts, two problems surfaced and were fixed:

1. **QEMU `Bus 'ahci0.2' not found`** (my error). I had assumed a single
   `ich9-ahci` controller exposes ports `ahci0.0..0.2`; the port count
   varies by QEMU version and `ahci0.2` was not present. Fixed by giving
   each optical device its **own** AHCI controller (`ahci1`, `ahci2`),
   whose port `.0` always exists — no port-count assumption. Disk = ahci0,
   boot ISO = ahci1, secondary ISO = ahci2; distinct controllers, so they
   coexist without collision in every target.

2. **ECAM frame leak → out-of-memory loop** (my bug; root cause found on
   the second run's logs). The ECAM accessor obtained kernel virtual space
   with `vmm_alloc_kernel_pages(256)` — which *also allocates 256 real RAM
   frames* — and then remapped that range to ECAM MMIO with `vmm_map_page`,
   orphaning the 256 RAM frames (they stay marked allocated forever). With
   the LRU cache re-allocating windows on every hotplug bus rescan, this
   leaked 1 MB of RAM per window per rescan, unbounded, until the frame
   pool was exhausted (`kpages_alloc n=256` failing in a loop). The
   exhausted/foamed allocator then mis-answered `pmm_is_frame_allocated`,
   so drivers were wrongly denied their MMIO BARs (e.g. AHCI at
   `0xfebd5000`), cascading into every PCI driver failing to start.

   The correct pattern is the one `map_lapic_mmio`/`map_ioapic_mmio`
   already use: reserve the VA, **immediately free the backing RAM frames**
   the allocator attached, then map the MMIO with `paging_map_page`. The
   fix gives each of the `ECAM_BUS_CACHE` (8) LRU slots a *permanent* 1 MB
   VA window allocated once (frames freed up front); eviction merely
   repoints that window's page-table entries to a different bus's MMIO.
   The MMIO address is therefore never routed through `kpages_free` /
   `pmm_free_frame`, no RAM frame is ever consumed by ECAM, and the
   footprint is a fixed 8 MB of VA regardless of how many buses are
   probed. The same frame-leak pattern in `scratch_map` (used to read
   firmware tables at boot) was fixed the same way.

   Note on the earlier `n=65536`: that was a *different* number seen on a
   build *before* this fix, when the accessor mapped a region's entire
   256-bus window at once. The bounded LRU (fix from the prior round)
   reduced it to `n=256` per window; this round removes the leak entirely.
   A separate pre-existing 256 MB allocation reported on the *legacy*
   precedent build (which has no ECAM code at all) is unrelated to ECAM and
   not yet diagnosed — it would need its own targeted logging.

3. **Device BARs denied as "allocated RAM"** (pre-existing latent bug,
   exposed by the q35 switch). With the ECAM leak fixed, the PCI scan
   completed but every MMIO driver (BGA video VRAM at `0xFD000000`, AHCI
   BARs at `0xFEBDxxxx`) was refused by the mmap syscall with "tried to map
   allocated RAM". Cause: `pmm_init` computed `total_frames` — and thus
   `pmm_get_ram_top()` — from the highest address of *any* memory-map
   entry, including reserved/MMIO regions. On q35 the map contains high
   reserved entries (ECAM at `0xB0000000`, BIOS flash near `0xFFFC0000`),
   so `ram_top` became ~4 GB and the syscall's "is this physical page
   allocated RAM?" guard treated device BARs as RAM. Fixed by deriving
   `total_frames` only from `AVAILABLE` entries (actual usable RAM), which
   is the same filter the frame-marking loop already uses. On the legacy
   `pc` machine the map had no such high entries, which is why the bug only
   surfaced after switching to q35.

4. **Disk invisible: AHCI driver is single-instance, q35 had it on the
   wrong controller** (QEMU-config issue, mine). With the BAR mapping fixed
   the AHCI driver finally ran, but the disk utility still saw no install
   target. Cause: the MainDOB AHCI driver is single-instance (the log shows
   later hotplug spawns exiting with "already running; second instance
   exits"), while my earlier QEMU config put the disk on its own controller
   (ahci0) and the boot CD on a *separate* controller (ahci1). The one
   surviving driver instance attached to whichever controller hotplug
   reached first and enumerated only that one — it found the optical drive,
   not the disk. Fixed by putting every SATA device on the SAME controller
   (ahci0) on distinct ports: disk on ahci0.0, boot/empty/secondary optical
   on ahci0.1 (and ahci0.2 for run-cd's third device). The driver's own
   port scan confirmed ich9-ahci exposes ports .0..5, so this is safe, and
   it matches the real ICH8 target (one AHCI controller, several ports).

## Known open issue (needs the next run's symbol resolution)

After the AHCI driver reports "N device(s) online" and surfaces the optical
drive to hotplug as a subdevice, the kernel takes a page fault *on behalf
of* the ahci process:

    [FAULT] Kernel Page Fault on behalf of 'ahci' (PID 17)
    [FAULT]   CR2: 0xc18506bc  EIP: 0xc010fd6d  not-present read

This is in code the driver only reaches now that the BAR mapping works (in
prior builds it died earlier), so it was never exercised before. The fault
is a kernel-address read triggered by a syscall the driver made in its
post-init path (sending the subdevice IPC message, or DMA/partition setup).
The IRQ-sharing syscalls added this session were reviewed and are
bounds-safe; the IPC path was not modified this session. Resolving it needs
the faulting symbol, which the build already emits:

    i686-elf-addr2line -e build/kernel.bin -f 0xc010fd6d
    i686-elf-nm -n build/kernel.bin | awk '$1 <= "c010fd6d"' | tail -3

The `[IRQ] line 10/11: irq_done timeout ... force-unmasking` lines just
before the fault are NOT errors — that is the IRQ-sharing safety timer
working as designed (a driver registered a line but never reported done, so
the line is force-unmasked rather than left wedged). Benign.

## AHCI driver: hardening + runtime disk hotplug (this session)

The AHCI driver (`drivers/ahci/main.c`) had never been exercised before the
porting work got it running. Reviewed and changed against two requirements
(multiple drives; disks appearing/disappearing at runtime):

Correctness for multiple drives:
- **Per-port command serialization.** `issue_cmd`/`issue_atapi` use command
  slot 0 and a single cmd_table/dma_buf per port, so two concurrent commands
  on the *same* port would clobber each other. Added a one-deep `cmd_busy`
  lock per port, taken/released under `enter_critical`; a second request to
  a busy port returns a retryable error. Different ports stay fully
  concurrent (all state is per-port).
- **Per-port reply buffers.** The single global `reply_buf` was replaced
  with `reply_buf[MAX_PORTS][...]`, so a reply's payload for one port is not
  overwritten by a concurrent read on another.

Runtime disk hotplug (the requirement the old code explicitly denied — it
exited when no disk was present at boot and only scanned ports once):
- **PhyRdy-change interrupt (PxIE.PRCS, bit 22)** is now enabled on every
  implemented port — both populated ports (so removal is caught) and empty
  ports (so insertion is caught). The PxSERR.DIAG.N latch is cleared on each
  event (writing PxIS alone does not clear it, or the IRQ re-asserts).
- **Dispatcher** detects PRCS and forwards the port number to a new
  **hotplug thread** (the COMRESET/bring-up work is blocking and must not
  run in the dispatcher, which has to stay free to deliver the IDENTIFY
  completion).
- **Hotplug thread** debounces, re-reads SSTS, and reconciles: link up with
  no prior device → `port_setup` + announce (partition scan for a disk,
  subdevice for optical); link down with a prior device → retract
  (delta-based `partition_scan_announce` with a failing reader emits
  SUBDEVICE_GONE for the disk's partitions; `ahci_announce_optical(false)`
  for optical) + `port_stop`.
- **Stay resident with zero disks.** The old "release the bubble and exit if
  port_count == 0" path was removed; an empty controller now stays alive to
  catch a later insert.
- **DMA reuse.** There is no `dma_free` syscall, so per-port DMA is allocated
  once and retained across removal/re-insert (`port_alloc_dma` is idempotent;
  teardown and `port_setup` preserve the DMA pointers). This bounds DMA to
  one set per physical port and avoids a per-insert leak.

Completion ports are now created for every *implemented* port (not just
those present at boot), since a port that gains a disk at runtime needs one
for the IDENTIFY issued during bring-up. The optical media-change thread is
likewise always started, so a runtime-inserted optical drive is fully
supported.

Status: compiles cleanly (native `-Wall -Wextra`, zero warnings). Untested
at runtime like everything else here. The kernel page fault above is
independent of these changes and still needs symbol resolution.

## AHCI driver crash (PID 17) — root cause + fix; storm note

Boot log on QEMU showed "[hotplug] Bubble #3: driver crashed (PID 17)" with
two AHCI controllers present (00:04.0 and 00:1f.2, both 8086:2922), plus a
"[sata-storm]" spurious-IRQ counter climbing on port 0 (n=201, 401, ...).

Root cause of the crash (CONFIRMED, not a guess): the AHCI main() called
dob_driver_attach() BEFORE dob_server_init(). dob_driver_attach sends
HOTPLUG_READY(dob_server_get_port()), but server_port is a static = 0 until
dob_server_init runs — so it announced port 0 to hotplug, which uses that
port to identify the reporting driver and to route the device reply. With a
bogus port the reply mis-routes and the (duplicate) instance crashes, non-
deterministically (uninitialised-value signature: PID 17 crashed, PID 18
sometimes survived). Proof it's an ordering bug: ac97 and bga both do
dob_server_init -> dob_server_get_port -> dob_driver_attach, and
hotplug_driver.h explicitly says "Call AFTER dob_server_init." AHCI was the
only driver violating it. The bug surfaced only with TWO controllers (the
multi-controller / "pluralism" case): hotplug spawns one instance per
controller and the duplicates crashed on the premature attach before
reaching the duplicate guard.

Fix (drivers/ahci/main.c main(), reordered to match ac97/bga):
  1. Duplicate guard FIRST using the registry only (dob_registry_find("ahci")),
     no dob_server_get_port() — a duplicate calls dob_driver_released() (a
     no-op if it has no bubble) and _exit(0). No crash.
  2. dob_server_init("ahci") next, so the port is valid.
  3. dob_driver_attach(&dev) after, with a valid port.
  4. Standalone no-controller exit now calls dob_registry_unregister("ahci")
     before _exit, closing a window (which this reorder would otherwise have
     opened) where an IDE-only machine leaves 'ahci' registered to a dead
     port and DobFileSystem mis-mounts an ATA disk over the AHCI protocol.

Storm note (NOT yet fixed — deliberately): the [sata-storm] counter flags
IRQs on port 0 with cmd_inflight==0 and CI=0, i.e. spurious interrupts with
no command outstanding, on a port whose TFD/CMD look healthy. Leading
hypothesis is the two controllers sharing an INTx line (each dispatcher sees
the shared IRQ and finds nothing in flight on its own port 0). BUT the crash
fix substantially changes duplicate handling, so the storm must be RE-
MEASURED on a build with this fix before theorising further: if only the
legitimate instance remains and the storm is gone, it was the duplicate/
collision; if it persists with a single active driver, it's an independent
shared-IRQ / spurious-D2H issue to handle separately. No speculative storm
change was made. Compiles clean; untested.

## "driver crashed" (PID 17) was cosmetic — race in clean RELEASE; fixed

After the attach/init reorder, the storm dropped to n=1 and duplicates print
"second instance exits", but hotplug still logged "Bubble #3: driver crashed
(PID 17)". Investigation showed this is NOT a real fault: it's a race in the
clean-release path, and the system is fully functional (disk mounted, boot
role claimed, partition icon shown).

Mechanism: hotplug is single-threaded — its loop does receive -> handle ->
bubble_reap_dead() each iteration. The duplicate AHCI instance calls
dob_driver_released() (synchronous SYS_SEND; hotplug replies because the loop
replies to any type==1 message) then _exit(0). handle_driver_released() did
kill(pid) THEN state=DEAD. If the duplicate's process became dead while the
bubble was still LIVE/ATTACHING (it also _exits on its own), the reaper saw
"dead process + LIVE bubble" and misreported it as a crash.

Fix (boot/hotplug/main.c, handle_driver_released): set b->state = BUBBLE_DEAD
BEFORE kill(b->driver_pid). Single-threaded + reaper-after-each-message means
the reaper can then only observe LIVE-and-alive (no-op) or DEAD (clean reap),
never LIVE-and-dead. A clean RELEASE is no longer logged as a crash. (Safe
because driver_pid is read before any reap runs; and killing the duplicate
while it is a blocked synchronous sender is safe thanks to the earlier
ipc_cleanup_thread fix that unlinks port_node from pending_senders.)

Status of SATA multi-controller ("pluralism"): crash fixed (real, via attach/
init reorder), storm fixed (was the duplicate collision), cosmetic crash log
fixed (this race). Empty ports now arm hotplug listen. Remaining: optical
media hotplug on QEMU needs TUR polling (SSNTF=0, no async notification), and
real second-HDD hotplug should be verified. All changes compile; untested.

## "driver crashed" persisted — real cause: death in ATTACHING, not the kill order

The BUBBLE_DEAD-before-kill reorder did NOT remove the "Bubble #3: driver
crashed (PID 17)" line. Re-analysis of the (reordered) AHCI main() shows why:
the duplicate guard now runs BEFORE dob_driver_attach, so a duplicate exits
WITHOUT ever sending HOTPLUG_READY. Hotplug therefore keeps the bubble in
ATTACHING (it only goes LIVE on READY). The duplicate sends HOTPLUG_RELEASED
then _exit(0); if its process dies before hotplug's single-threaded loop
processes that RELEASED, handle_driver_released() is never reached for it, so
the kill-order reorder is irrelevant. The reaper then sees an ATTACHING bubble
whose process is dead, finds no other DAS candidate, and logs "driver
crashed".

Correct fix (boot/hotplug/main.c, bubble_reap_dead): distinguish death in
ATTACHING from death in LIVE. A process that dies while still ATTACHING
(never sent READY) with no further candidate did not crash a working driver —
it bowed out during bring-up. The canonical case is a singleton driver like
AHCI: hotplug spawns one instance per matching controller, the extras exit at
their duplicate guard while the bubble is still ATTACHING. That is now logged
as "driver exited during attach ... bubble retired" and the bubble is retired
cleanly. Only a death in LIVE state is still reported as a genuine crash. This
is robust to the RELEASED-vs-_exit race because it classifies on bubble state,
not on message timing. (The earlier DEAD-before-kill reorder is kept; it makes
the handle_driver_released path consistent when RELEASED *is* processed.)

Tradeoff: a driver that genuinely crashes DURING bring-up (rare) is now
labelled "exited during attach" rather than "crashed"; functionally identical
(bubble retired, hardware freed), just less alarming wording. Deaths of LIVE
drivers — the real crashes — are unaffected. Compiles; untested.

## Optical CD/DVD hotplug on QEMU — TUR polling fallback (priority feature)

QEMU's ich9-ahci reports CAP.SSNTF=0 (boot log: "[sata-media] port 5: ...
SSNTF=0 SDBS-enabled=0"), so the Set Device Bits FIS async-notification
interrupt is never enabled and the dispatcher never posts a media event;
ahci_media_thread would block forever and inserting/ejecting a disc at
runtime is invisible. (On real silicon with SSNTF the interrupt path works
and this whole fallback is dormant.)

Fix (drivers/ahci/main.c), reusing ALL existing detection logic — no
duplication:
  - media_poll_needed (global): set in port_setup's optical branch when the
    controller lacks SSNTF.
  - ahci_media_poll_thread: spawned in init only if media_poll_needed. Every
    AHCI_MEDIA_POLL_MS (1500 ms) it posts {code=2, arg0=port} to
    media_event_port for each optical port — the SAME event the SDBS
    interrupt would have produced. ahci_media_thread then runs its normal
    ahci_media_present() (TUR + REQUEST SENSE), debounces against
    media_present[p], and calls ahci_announce_optical on a real transition.
    The media thread is oblivious to whether the event came from the
    dispatcher or the poller.
Safety: ahci_media_present touches only the optical port and takes the
per-port cmd_busy lock, so it never interferes with the boot disk on port 0.
If the media thread is briefly busy (worst-case ~5 s on unit-attention
churn), queued poll events are harmless — debounce makes reprocessing
idempotent (no duplicate announce). Compiles clean; UNTESTED — needs a boot
on QEMU and inserting/ejecting a disc via the GUI Removable-media menu to
confirm the icon appears/disappears.

## USB hotplug — device detection + DAS match (UHCI), shared matcher

Goal: when a device is plugged into a USB port, the host-controller driver
enumerates it, matches its USB class triple against the device-level DAS
files in /SYSTEM/CONFIG/DAS/USB/, and emits HOTPLUG_SUBDEVICE_APPEARED so the
existing subdevice pipeline (hotplug -> desktop icon -> click -> DAS action)
draws the icon. Same model as AHCI partitions; no new icon/action machinery.

State found: usb_uhci already had a complete enumeration state machine
(ST_PORT_RESETTING -> ST_SETADDR_SENT -> ST_DEVDESC_SENT -> ST_CONFDESC_SENT
-> ST_ENUM_DONE) that reads device + interface descriptors. It logged the
class triple ("BLOCK 2: report only") and explicitly marked where "Block 3"
(DAS match + announce) should go. usb_ehci/usb_xhci are far less complete
(enumeration not at this level yet).

Added:
  - drivers/usb_common/usb_das.{h,c}: a SHARED USB-DAS matcher (no hardware,
    no IPC). Reads every *.das in /SYSTEM/CONFIG/DAS/USB/ via the
    DobFileSystem stub, parses key=value (usb_class/subclass/protocol,
    match_on interface|device, subdev_class/subdev_subclass, label), matches
    the enumerated device's triple, and returns the most specific match
    (non-wildcard fields beat wildcards; 0xFF = any). Designed for reuse by
    usb_ehci/usb_xhci.
  - usb_uhci Block 3: at ST_ENUM_DONE, fills usb_das_device_t from enum_dev/
    enum_iface, calls usb_das_match(), and on a hit emits
    HOTPLUG_SUBDEVICE_APPEARED with the DAS's subdev class/subclass and
    provider_service="usb_uhci", provider_token=port. Logs the match.
  - drivers/Makefile: dedicated usb_uhci.mdl rule that compiles
    usb_common/usb_das.c and links DobFileSystem_stub.o (for dobfs_List/
    Open/Read/Close). The existing config/DAS/USB/mass_storage.das already
    defines the mass-storage signature (usb 0x08/0x06/0x50 -> subdev
    0x01:0x06).

Not done / next: removal path (SUBDEVICE_GONE when the device is unplugged)
is not wired yet; EHCI/xHCI still need their enumeration brought up before
they can reuse the matcher. All four files compile clean; UNTESTED. On QEMU
the piix3-usb is UHCI, and the existing "[uhci] ENUM DONE ..." log plus the
new "[uhci] DAS match ..." line will show whether detection + match work
when a USB device is attached.


## USB test: "no device assigned" — driver raced ahead of hotplug; fixed

First run-usb boot: the standalone piix3-usb-uhci controller (8086:7020,
cls 0c:03:00) WAS recognised and hotplug spawned usb_uhci.mdl (PID 18), but
the driver immediately logged "[uhci] no device assigned by hotplug" and
exited, so no enumeration and no icon.

Cause: usb_uhci called dob_server_init -> dob_driver_attach with NO
dob_registry_wait("hotplug", ...) in between. hotplug spawns its drivers
during the PCI scan, BEFORE entering its service loop, so the synchronous
HOTPLUG_READY from the driver can arrive before hotplug is ready to answer;
dob_driver_attach then returns false and the driver exits. The other
hotplug-spawned drivers (bga line 2695, ac97 line 1065) all
dob_registry_wait("hotplug", 5000) before attach for exactly this reason.
usb_uhci was the only one missing it.

Fix: added dob_registry_wait("hotplug", 5000) before dob_driver_attach in
usb_uhci main(), matching bga/ac97. Compiles; retest with `make run-usb` —
expect [uhci] ENUM DONE + [uhci] DAS match + a USB icon.

## USB: driver went LIVE but never enumerated boot-present device; fixed

With the registry_wait fix, usb_uhci now goes LIVE, inits hardware
([uhci] io_base=0xc580 irq=10 ...), and enters Global Suspend — but then
nothing: no ENUM DONE, the boot-attached pendrive is never enumerated.

Cause: the driver was designed purely around the resume-IRQ model (its own
header says so): in Global Suspend it waits for a resume interrupt, which the
UHCI controller raises on a CONNECT/DISCONNECT *change* while suspended. A
device already attached at boot produces no such change, so the driver read
port_present[p]=1, stored it, slept, and waited forever for an event that had
already happened. Same shape as the optical-media case: a mechanism waiting
for an event a static-at-boot device never generates.

Fix (drivers/usb_uhci/main.c, uhci_init_hw tail): after seeding
port_present[], if any port already has a device (boot_port >= 0), enumerate
it immediately instead of suspending. reset_and_setup leaves the controller
STOPPED (RS=0), and the resume path normally runs it via uhci_leave_suspend
before enumerating, so the boot path now does the equivalent: enable the
completion interrupts the FSM needs (IOC|SHORT|TIMEOUT) and set RS=1, then
enum_begin(boot_port). If no device is present at boot, it enters suspend as
before to await a hot-insert resume IRQ. Compiles; retest with make run-usb.

Expected now: [uhci] device present at boot on port N; enumerating ->
[uhci] enumerating port N (reset asserted) -> ENUM DONE -> DAS match -> icon.
If the FSM stalls mid-enumeration on QEMU, the last [uhci] line printed tells
us which step (reset / SET_ADDRESS / GET_DESCRIPTOR) to look at.


## ATA: real DMA->PIO fallback (Compaq Armada E500 — no partitions shown)

Symptom (physical Compaq Armada E500, mSATA SSD via IDE adapter): DobDisk
reads the disk fine — capacity, model, total sectors, and the entire SMART
attribute page all display correctly — but the partition list shows "0 voci"
with no free space either. Same with a previous Windows 98 disk that
definitely had a valid MBR. On QEMU the same code shows partitions.

Reasoning (no serial console on the physical machine, so this is inferred
from what IS visible): SMART READ DATA and IDENTIFY are PIO commands and they
work, so PIO is healthy. DobDisk's MBR load does block_read(disk, lba=0, 1);
if that returns false, load_mbr returns early, cur_mbr_valid stays false, and
rebuild_rows() emits no rows at all — exactly "0 voci, no free space". Slot-0
sector reads use DMA (BMIDE), not PIO. A modern SSD behind an IDE adapter
advertises UDMA, so dma_ok comes up true, but the Armada-era bus-master IDE
may not actually transfer — and the old ata_read had a logic hole: it tried
DMA, and only did PIO when dma_ok was ALREADY false. A runtime DMA failure
(dma_ok still true) skipped PIO, just ran ata_recover and retried DMA, so it
never fell back. The promised "PIO fallback" only covered boot-time DMA
absence, not runtime DMA failure.

Fix (drivers/ata/main.c, ata_read + ata_write): on a DMA failure, fall
through to PIO IN THE SAME attempt instead of only retrying DMA; count
consecutive DMA failures and, after DMA_FAIL_LIMIT (3), set dma_ok=false to
run PIO-only for this disk from then on (a broken bus-master will never
start working, so stop wasting time on it). A successful DMA resets the
streak. On QEMU dma_xfer never fails, so DMA is kept and nothing degrades.
LBA28 PIO addresses the whole 62,533,296-sector (~29.8 GB) disk, so the
fallback is fully sufficient here.

Self-diagnosing: if DMA was the cause, partitions/free space now appear on
the Compaq; if it STILL shows 0 voci, the cause is elsewhere (MBR
parse/geometry) and we have ruled DMA out without harm. Compiles; untested on
hardware.
